# WorldBox Studio AI — 기술 스택 및 시스템 아키텍처

> 인터랙티브 AI 샌드박스 플랫폼의 기술 선정 근거, 전체 아키텍처, DB 설계, 시스템 프롬프트 자동화 전략을 정리한 문서입니다.

## 1. 기술 스택 선정 (Tech Stack)

복잡한 상태 관리(에디터, 마인드맵), 실시간 인터랙션(채팅), PWA 기반 크로스 플랫폼 지원을 목표로 아래 스택을 채택합니다.

### 1.1 프론트엔드 (Web / PWA & App)

| 항목 | 선정 기술 | 선정 근거 |
|------|-----------|-----------|
| **프레임워크** | **SvelteKit** + **Tailwind CSS** | 반응성 우수, 번들 크기 작음 → PWA에서 네이티브 앱에 가까운 체감 속도 |
| **마인드맵 UI** | **Svelvet** (커스텀) | 캔버스 기반 노드/엣지 UI 구현에 적합 |
| **향후 확장** | API 기반 분리 | Flutter 웹뷰/네이티브 UI로 마이그레이션 시 핵심 로직 재사용 용이 |

### 1.2 백엔드 (API & 비즈니스 로직)

| 항목 | 선정 기술 | 선정 근거 |
|------|-----------|-----------|
| **프레임워크** | **Django** + **Django REST Framework (DRF)** | 세계관·폴더·캐릭터 간 복잡한 관계를 ORM으로 설계·관리하기에 생산성 높음 |
| **비즈니스 로직** | Django 앱/서비스 레이어 | 메타데이터·듀얼 스탯 → JSON 시스템 프롬프트 변환 등 백그라운드 로직 구현에 적합 |

### 1.3 데이터베이스 & 스토리지

| 항목 | 선정 기술 | 선정 근거 |
|------|-----------|-----------|
| **RDB** | **PostgreSQL** | 캐릭터 기본 정보는 정규화된 테이블, 특수 스킬·메타데이터는 **JSONB**로 유연 보관 |
| **오브젝트 스토리지** | **MinIO** | S3 호환, 세계관 레퍼런스 이미지·아바타 등 에셋 저장용 |

### 1.4 AI 및 RAG 파이프라인

| 항목 | 선정 기술 | 선정 근거 |
|------|-----------|-----------|
| **오케스트레이션** | **LangChain** (또는 LlamaIndex) | RAG·메모리·LLM 호출 파이프라인 구성 |
| **벡터 DB** | **ChromaDB** (또는 Milvus) | 장기 기억·관계도 기반 제3자 인식을 위한 텍스트/관계 벡터 저장·검색 |
| **LLM** | **OpenAI / Gemini** | 최종 롤플레잉·내러티브 텍스트 생성 |

## 2. 전체 시스템 아키텍처

시스템은 **클라이언트 → API/비즈니스 → AI 인퍼런스 → 데이터** 4개 레이어로 구분합니다.

```
┌─────────────────────────────────────────────────────────────────┐
│  Client Layer (SvelteKit PWA)                                     │
│  ├─ Editor UI                                                    │
│  │   ├─ 노코드 페르소나 에디터 (메타데이터, 듀얼 스탯 차트)        │
│  │   └─ 마인드맵 (캐릭터 관계도)                                  │
│  └─ Player UI                                                    │
│      ├─ 대화형 스토리 에디터 (채팅)                               │
│      ├─ 스마트 멘션 (@캐릭터 난입)                                │
│      └─ 스킬/액션 버튼                                           │
└─────────────────────────────────────────────────────────────────┘
                                    │
                                    │  REST API / WebSockets
                                    ▼
┌─────────────────────────────────────────────────────────────────┐
│  Backend Layer (Django + DRF)                                    │
│  ├─ World/Project Manager  → 폴더·파일 계층 구조 관리             │
│  ├─ System Prompt Builder  → 시각적 데이터 → JSON/XML 프롬프트   │
│  └─ TRPG Game Master Engine → 마력 소모, 스킬 성공/실패 계산      │
└─────────────────────────────────────────────────────────────────┘
                                    │
                                    │  LangChain / LlamaIndex
                                    ▼
┌─────────────────────────────────────────────────────────────────┐
│  AI & RAG Layer                                                  │
│  ├─ Vector DB (ChromaDB)  → 세계관 배경, 장기 기억 컨텍스트 검색  │
│  ├─ Graph DB (Optional)   → 마인드맵 관계도 추론 연동             │
│  └─ LLM API (Gemini/OpenAI) → 롤플레잉 텍스트 생성               │
└─────────────────────────────────────────────────────────────────┘
                                    │
                                    ▼
┌─────────────────────────────────────────────────────────────────┐
│  Data Layer                                                      │
│  ├─ PostgreSQL  → 유저, 세계관, 캐릭터 명세, 채팅 로그            │
│  └─ MinIO       → 이미지, 프로필·에셋 스토리지                    │
└─────────────────────────────────────────────────────────────────┘
```

### 2.1 레이어별 역할 요약

| 레이어 | 역할 |
|--------|------|
| **Client** | 창작(에디터)·플레이(채팅, @멘션, 스킬) UX, PWA로 설치·오프라인 대비 |
| **Backend** | 세계관/캐릭터 CRUD, 시스템 프롬프트 조립, TRPG 규칙(스탯·스킬) 처리 |
| **AI & RAG** | 벡터 검색으로 컨텍스트 보강, (선택) 그래프 기반 관계 추론, LLM 호출 |
| **Data** | 영구 저장: RDB(구조화 데이터), MinIO(파일/에셋) |

## 3. 핵심 모듈별 DB 설계 전략

마인드맵 기반 관계도와 노코드 시스템 프롬프트 변환을 지원하는 Django 모델 골격입니다.

### 3.1 ER 개요

- **World** : 프로젝트(세계관) 최상위 컨테이너  
- **Character** : 한 World 내 캐릭터, 스탯·메타데이터는 JSONB  
- **Relationship** : Character ↔ Character 관계 (스승/제자, 짝사랑, 적대 등)  
- **Skill** : 캐릭터별 TRPG 스킬 (마나 소모, 시스템 효과 등)

### 3.2 Django 모델 예시

```python
# Django Models (핵심 엔티티)

from django.db import models

class World(models.Model):
    """세계관 최상위 컨테이너"""
    name = models.CharField(max_length=100)  # 예: '에버그린데일'
    description = models.TextField(blank=True)
    # created_at, updated_at, owner 등은 공통 Mixin으로 관리 권장


class Character(models.Model):
    """노코드 페르소나 데이터"""
    world = models.ForeignKey(World, on_delete=models.CASCADE)
    name = models.CharField(max_length=100)

    # 듀얼 스탯 시스템 (JSONB)
    # 예: {"str": 80, "agi": 90, "spirit": 70, "mana": 300, "leadership": 50}
    stats = models.JSONField(default=dict)

    # 심화 메타데이터 (선호 음악, 소중한 물건, 자주 가는 곳 등)
    metadata = models.JSONField(default=dict)


class Relationship(models.Model):
    """관계도 (그래프 구조를 RDB로 표현)"""
    source_character = models.ForeignKey(
        Character, related_name="source_relations", on_delete=models.CASCADE
    )
    target_character = models.ForeignKey(
        Character, related_name="target_relations", on_delete=models.CASCADE
    )
    relation_type = models.CharField(max_length=50)  # '스승과 제자', '적대관계', '짝사랑' 등
    weight = models.IntegerField(default=1)  # 관계 깊이/강도

    class Meta:
        unique_together = [("source_character", "target_character", "relation_type")]


class Skill(models.Model):
    """TRPG 액션/스킬 시스템"""
    character = models.ForeignKey(Character, on_delete=models.CASCADE)
    name = models.CharField(max_length=100)  # 예: '달빛 베기'
    mana_cost = models.IntegerField(default=0)
    system_effect = models.TextField()  # 시스템 프롬프트에 주입될 스킬 효과 설명
```

### 3.3 설계 포인트

- **Character.stats / metadata**  
  기초·전투 스탯, 취향·소품 등 유동 필드는 JSONB로 두어 스키마 변경 없이 확장.
- **Relationship**  
  마인드맵에서 편집한 관계를 그대로 저장하고, 시스템 프롬프트 빌드 시 `source_relations`/`target_relations`로 조회.
- **Skill**  
  채팅 시 스킬 버튼 클릭 → 백엔드에서 마나/스탯 검사 후 `system_effect`를 프롬프트에 삽입.

## 4. 시스템 프롬프트 자동화 (실행 전략)

기획서의 **시각적 데이터 → 시스템 프롬프트 자동 변환**은 백엔드에서 **팩토리/빌더 패턴**으로 구현합니다.

### 4.1 트리거 시점

- 사용자가 **채팅방 입장** 시  
- 또는 **@멘션으로 캐릭터 소환** 시  

해당 캐릭터(및 필요 시 관계/스킬)를 기준으로 시스템 프롬프트를 조립합니다.

### 4.2 실행 흐름

1. **데이터 Fetch**  
   `Character`, `Relationship`, `Skill` (및 필요 시 `World`)를 DB에서 로드.
2. **Prompt 빌드**  
   아래와 같이 `build_system_prompt(character)`로 문자열(또는 JSON) 생성.
3. **LLM 주입**  
   생성된 프롬프트를 LangChain의 `SystemMessage` 등에 넣어 롤플레잉 세션을 시작.

### 4.3 프롬프트 빌더 예시 코드

```python
def build_system_prompt(character: Character) -> str:
    """캐릭터의 스탯, 관계, 스킬을 반영한 시스템 프롬프트 생성"""

    # 1. 기본 배경 및 스탯 주입
    world_name = character.world.name
    stats = character.stats or {}
    base_prompt = (
        f"당신은 '{world_name}' 세계관의 {character.name}입니다. "
        f"당신의 능력치(지능, 마력 등)는 {stats}에 따라 행동과 말투에 반영됩니다.\n\n"
    )

    # 2. 관계도 데이터 주입 (RAG/컨텍스트)
    relations = Relationship.objects.filter(source_character=character).select_related(
        "target_character"
    )
    relation_prompt = "당신의 인간관계는 다음과 같습니다:\n"
    for rel in relations:
        relation_prompt += f"- {rel.target_character.name}: '{rel.relation_type}' (강도: {rel.weight})\n"
    relation_prompt += "\n"

    # 3. TRPG 스킬/액션 세팅
    skills = Skill.objects.filter(character=character)
    skill_prompt = (
        "플레이어가 아래 기술을 사용할 때, 조건에 맞게 화려하게 묘사하세요:\n"
    )
    for skill in skills:
        skill_prompt += f"- {skill.name} (마나 소모: {skill.mana_cost}): {skill.system_effect}\n"

    return base_prompt + relation_prompt + skill_prompt
```

### 4.4 LLM 연동 (개념)

```python
# LangChain 예시 (의사 코드)
from langchain_core.messages import SystemMessage

system_message = SystemMessage(content=build_system_prompt(character))
# 채팅 체인에 system_message 주입 후 유저 메시지로 대화 진행
```

## 5. 배포: Docker

서비스 배포는 **Docker** 기반으로 수행한다. 개발·스테이징·운영 환경의 일관성과 이식성을 확보한다.

### 5.1 배포 스택

| 항목 | 선정 | 비고 |
|------|------|------|
| **컨테이너화** | **Docker** | 이미지 빌드·실행 단위 |
| **오케스트레이션** | **Docker Compose** | 로컬·단일 서버 배포 시 프론트·백엔드·DB·스토리지·AI 서비스 일괄 구성 |

### 5.2 서비스 구성 (Compose 예시)

- **frontend**: SvelteKit 빌드 결과를 Nginx 등으로 서빙 (또는 Node 서버). PWA 정적 에셋 포함.
- **backend**: Django + Gunicorn(uWSGI). DRF API, 시스템 프롬프트 빌더, TRPG GM 엔진.
- **postgres**: PostgreSQL. 영구 볼륨으로 DB 데이터 유지.
- **minio**: MinIO. 이미지·에셋 스토리지.
- **chroma** (선택): ChromaDB. 벡터 저장·RAG 검색.
- **LLM**: 외부 API(OpenAI/Gemini) 사용 시 별도 컨테이너 불필요; 백엔드에서 환경 변수로 API 키 설정.

이미지 빌드·환경 변수·볼륨·네트워크는 `docs/기술_스택_및_아키텍처.md` 및 프로젝트 내 `docker-compose*.yml`, `Dockerfile` 규약을 따른다.

### 5.3 원칙

- 배포 가능한 변경은 Dockerfile·Compose 수정으로 반영하고, 문서에 필요한 경우 “배포 절차”를 갱신한다.
- 시크릿(DB 비밀번호, LLM API 키 등)은 환경 변수 또는 Docker secrets로 주입하며, 저장소에 평문으로 커밋하지 않는다.

## 6. 동시 접속·부하 대응 및 확장성

여러 사용자가 동시에 API를 사용할 경우를 전제로, 부하 분산·리소스 관리·확장 전략을 아래와 같이 둔다.

### 6.1 부하가 발생하는 구간

| 구간 | 특성 | 대응 방향 |
|------|------|------------|
| **DRF API** | 동시 요청 수 ↑ → 스레드/프로세스 포화 | 워커 수·비동기·수평 확장 |
| **PostgreSQL** | 커넥션 수·쿼리 부하 | 커넥션 풀, 읽기 분산(선택) |
| **LLM 호출** | 지연 큐·비용·타임아웃 | 비동기·큐·캐시(선택) |
| **RAG/ChromaDB** | 검색 요청 동시 다발 | 인덱스·캐시·재시도 |
| **WebSocket** | 채팅 세션 수 ↑ | 연결 수 제한·스케일 아웃 |

### 6.2 백엔드 (Django + Gunicorn)

- **워커 수**: Gunicorn `workers`(및 `threads`)를 CPU·메모리에 맞게 설정. 동시 요청을 여러 워커가 나눠 처리.
- **DB 커넥션**: `CONN_MAX_AGE` 또는 **PgBouncer** 등으로 PostgreSQL 커넥션 풀 사용, 커넥션 수 상한 유지.
- **LLM 호출**: 동기 호출이 길어지면 워커가 블로킹되므로, **비동기 처리** 또는 **메시지 큐(Celery, RQ 등)** 로 LLM 요청을 워커 밖에서 처리. 타임아웃·재시도 정책 명시.
- **Rate limiting**: DRF Throttling으로 IP·유저별 요청 빈도 제한. 남용·비용(LLM) 방지.

### 6.3 캐시 (선택)

- **Redis** 등으로 다음을 캐시할 수 있음: 세계관/캐릭터 메타 조회, 시스템 프롬프트 조각, RAG 검색 결과(같은 쿼리 재사용 시). TTL·무효화 전략을 두어 일관성 유지.

### 6.4 수평 확장

- **백엔드**: 동일 이미지로 컨테이너 복제(예: `docker-compose up --scale backend=3`). 앞단에 **Nginx/트래픽 로드밸런서**로 분산.
- **WebSocket**: 스티키 세션(같은 클라이언트 → 같은 백엔드) 또는 Redis 등으로 채널/메시지 공유해 다중 인스턴스에서도 채팅 연속성 유지.
- **DB/ChromaDB/MinIO**: 단일 서버로 시작해, 트래픽 증가 시 PostgreSQL 읽기 레플리카·ChromaDB/MinIO 스케일 정책 검토.

### 6.5 설계 원칙

- 동시 사용자 수가 늘어나도 **단일 병목(DB, LLM, 워커)** 을 의식해 설계한다.
- 무거운 작업(LLM 추론, 대량 RAG)은 **비동기·큐**로 분리하고, API는 가능하면 빠르게 응답(예: 202 수락 후 폴링/WebSocket으로 결과 전달).
- 모니터링(요청 지연, 에러율, DB/Redis 연결 수)과 알람을 두어, 부하 구간을 사전에 파악한다.
